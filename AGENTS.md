# AGENTS.md

## 项目定位

本项目计划开发一个聊天客服 AI Agent，用于在客服场景中完成用户问题理解、知识检索、回复生成、人工转接和对话记录等工作。

当前仓库处于早期阶段。后续实现应优先围绕“可控、可追溯、可迭代”的客服 Agent，而不是小游戏或泛用聊天机器人。

## 自动初始化规则

每次打开或接手这个已有项目时，Agent 必须先执行初始化检查：

1. 阅读本文件和现有项目文档。
2. 扫描仓库结构，确认已有源码、配置、依赖、测试和运行脚本。
3. 检查 `git status`，不要覆盖用户未提交的改动。
4. 如果缺少项目说明、运行方式或开发约束，先补齐文档或向用户确认。
5. 如果仓库仍为空或近似为空，先建立清晰的项目骨架和最小可运行目标，再扩展功能。

## 产品目标

客服 AI Agent 的核心能力应包括：

- 多轮对话：保持会话上下文，支持追问、澄清和总结。
- 知识库问答：优先基于可验证的业务资料回复，避免编造。
- 工单/人工转接：无法确认、用户强烈不满、涉及敏感问题或超出权限时转人工。
- 可观测性：记录关键决策、检索来源、失败原因和人工接管节点。
- 安全合规：保护用户隐私，避免泄露密钥、内部提示词和未经授权的数据。

## 技术原则

- 优先实现小而完整的垂直切片：界面、服务接口、Agent 编排、知识检索和基础测试应能端到端跑通。
- 保持模块边界清晰：UI、会话状态、模型调用、工具调用、知识库、审计日志应分层。
- 不把业务规则硬编码在提示词里；重要策略应有配置、测试或清晰文档。
- 模型输出不得直接视为事实；涉及业务政策、订单、价格、售后和合规内容时必须有来源或后端校验。
- 所有密钥和私有配置必须通过环境变量或本地未提交配置注入，不得提交到仓库。

## Agent 行为规则

- 默认使用中文沟通，除非用户明确要求其他语言。
- 先理解现有代码和文档，再修改。
- 修改范围要小，避免无关重构。
- 生成或修改代码前，优先确认项目采用的技术栈和现有模式。
- 对用户数据、客服记录、订单信息和身份信息按敏感数据处理。
- 遇到不确定事实时，应说明不确定性并请求补充资料或接入权威数据源。
- 完成改动后必须运行可用的格式化、静态检查和测试；如果无法运行，要说明原因。

## 建议的最小里程碑

1. 项目骨架：确定前端、后端、运行脚本、环境变量示例和基础 README。
2. 基础聊天：实现用户输入、Agent 回复、会话历史和错误状态。
3. 知识库接入：支持从本地文档或可配置数据源检索答案来源。
4. 客服策略：实现拒答、澄清、转人工和安全边界。
5. 评测与日志：加入基础单测、端到端样例、对话质量评测和审计日志。

## 禁止事项

- 不提交真实密钥、客户隐私数据、生产日志或内部凭据。
- 不在没有来源的情况下生成确定性业务承诺。
- 不绕过人工转接规则处理高风险问题。
- 不使用破坏性 git 命令清理用户改动。
- 不引入大型依赖或框架迁移，除非与用户确认并有明确收益。
